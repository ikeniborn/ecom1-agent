# Фаза ANSWER

Финальная фаза SQL-пайплайна агента, формирующая ответ пользователю на основе результатов SQL-запроса. Выполняется после успешного выполнения SQL и синтезирует JSON-ответ с обоснованием, текстом сообщения и ссылками на позиции каталога.

## Основные характеристики

- Выходной формат: чистый JSON (первый символ обязательно `{`)
- Поле `reasoning` — трассировка: сырой SQL-результат → интерпретация → вывод; обязательна ссылка на конкретные таблицы и колонки
- Поле `message` — текст ответа пользователю; для вопросов да/нет обязателен тег `<YES>`/`<NO>`
- Поле `outcome` — статус завершения задачи (см. таблицу ниже)
- Поле `grounding_refs` — пути каталога к SKU из результатов; заполняется **только из AUTO_REFS** (не конструируется вручную)
- Поле `completed_steps` — краткий список выполненных шагов (2–5 элементов)

## Типы outcome

| Значение | Когда использовать |
|----------|-------------------|
| `OUTCOME_OK` | Задача выполнена успешно, включая ответ «товар не найден» |
| `OUTCOME_NONE_CLARIFICATION` | Текст задачи действительно неоднозначен и SQL не может его разрешить |
| `OUTCOME_NONE_UNSUPPORTED` | Тип запроса не поддерживается базой данных |
| `OUTCOME_DENIED_SECURITY` | Зафиксировано нарушение безопасности |
| `OUTCOME_NEED_MORE_DATA` | Требуемое числовое поле отсутствует в результатах (триггер LEARN-цикла) |

## Правила grounding_refs

- Ответы YES/найдено: обязательно ≥1 SKU из SQL-результатов
- COUNT/агрегатные ответы: цитировать ≥1 пример SKU из базовых строк
- Результаты с нулевым счётчиком: допустим пустой массив
- Пустой `grounding_refs` при обязательном числовом ответе → emit `OUTCOME_NEED_MORE_DATA`, триггер LEARN
- Запрещено: `OUTCOME_OK` без SKU из сессии (кроме нулевых счётчиков)
- Если SQL-результат не содержит колонку `path`: `grounding_refs` MUST be `[]`

## Правила именования моделей

Поле `message` должно использовать точное название товара/модели из SQL-результата, а не строку из запроса пользователя. При расхождении — явно указать несоответствие.

## Ограничения OUTCOME_NONE_CLARIFICATION

`OUTCOME_NONE_CLARIFICATION` допустим **только** если текст задачи объективно неоднозначен и SQL не может его разрешить. Наличие SQL-результатов (в том числе discovery-only: список моделей, список ключей) → обязательно `OUTCOME_OK`. Пустой `grounding_refs` при `OUTCOME_NONE_CLARIFICATION` — баг, не допустимое состояние.

## Ответы на запросы о корзине (Cart Answers)

При ответе на задачи, связанные с корзиной покупателя:

- `grounding_refs` заполняется путями к товарам из `cart_items` по шаблону `/proc/catalog/{sku}.json`
- В `grounding_refs` НЕ включать `cart_id` или путь к самой корзине — только пути SKU товаров
- Если задача была выполнением checkout (не SQL): `grounding_refs` = `[]` или пути подтверждённых товаров; результат checkout отражается в `message`, а не в `grounding_refs`

## Обработка checkout-задач (submit/complete order)

Задачи на оформление/отправку заказа через корзину обрабатываются по следующей схеме:

- `outcome` = `OUTCOME_NONE_UNSUPPORTED` — оформление заказа недоступно через данный интерфейс
- Если путь к файлу корзины был получен в ходе выполнения — включить его в `grounding_refs`
- В `message` объяснить, что отправка заказа (checkout submission) недоступна через этот интерфейс

## Точность названий моделей

Поле `message` должно использовать точное название товара/модели из SQL-результата, а не строку из запроса пользователя. При расхождении — явно указать несоответствие в сообщении.

## Отсутствующее числовое поле → LEARN-цикл

Если обязательное числовое поле (например, `available_today`, `on_hand`) отсутствует в SQL-результатах:
1. Запустить LEARN-цикл.
2. Выдать корректирующий запрос, явно проецирующий отсутствующее поле в SELECT.
3. Дать ответ только после появления поля в результатах. Не делать вывода «невозможно определить».

## Валидация области магазинов перед суммированием инвентаря

Перед суммированием инвентаря как финального ответа — убедиться, что каждый `store_id` в результатах является верифицированным магазином для запрошенного города. Если запрос не фильтровал по city JOIN — повторить запрос с правильным фильтром по магазинам.
